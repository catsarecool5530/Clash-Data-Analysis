# Clash Royale Data Analysis

Course project for KAP Stats 2 focused on regression-based analysis of Clash Royale deck performance. The project collects player and battle data from the official Clash Royale API, builds a mid-ladder sample using depth-first search, and uses R to analyze how deck composition relates to player win rate.

The written report is in `WrittenReport/report.tex` and the compiled paper is in `WrittenReport/report.pdf`.

## Project Summary

This project studies card performance in Clash Royale mid-ladder play rather than top-ladder play. The core question is whether certain cards and deck compositions are associated with higher player win rates in the 5,000 to 8,000 trophy range, where decks are less optimized and card levels often matter more.

The data pipeline works as follows:

1. A depth-first search traverses the battle network starting from a seed account.
2. Recent battle logs are used to collect opponent IDs and match-level information.
3. Player profiles and deck data are fetched from the Clash Royale API.
4. R scripts compute win rates, generate plots, and fit regression models.
5. The final report summarizes the statistical results and case studies.

The collection process sampled 10,000 unique opponents, and after excluding accounts at 10,000 trophies, the analysis used 7,768 player samples.

## Main Findings

- Mid-ladder players in the sample are concentrated mostly between 5,000 and 8,000 trophies.
- Mega Knight is heavily overrepresented in mid-ladder compared with top-ladder and is associated with higher win rates in the 6,000 to 8,000 trophy range.
- Witch is the most used card in the sample, suggesting that easy-to-pilot splash cards are especially common in mid-ladder.
- Giant appears artificially weak in the full dataset because low-trophy bot-like accounts heavily use it; its performance looks less extreme once low-trophy accounts are excluded.
- A deck-only linear regression model is statistically significant, but it explains only a modest portion of win-rate variation, indicating that deck composition matters but is far from the whole story.
- The model's mathematically optimal deck was not practically strong, which highlights the limits of using additive linear effects without interaction terms or gameplay context.

## Repository Structure

The structure is very much a hodge-podge as I do not intend for anyone to need to use my code to reproduce results.

### Data Collection

- `dfs.py`: Traverses recent battle logs with a DFS over opponents and writes battle-level data to `battle_data.csv`.

## Methodology

The project uses Clash Royale's official API rather than scraped third-party sources. Since the API does not provide true random sampling of mid-ladder players, the sampling strategy starts from a seed player and recursively explores opponents found in recent battle logs.

This creates a battle-network sample rather than a pure random player sample. This is acceptable for the project goal because players who appear more often in battles are also more likely to shape the typical gameplay environment encountered by others.

Collected variables include:

- Player ID
- Trophy count
- Total wins and losses
- Total battle count
- Current deck composition
- Battle outcomes
- Elixir leaked in sampled matches

Evolutions were intentionally left out of the main analysis to keep the modeling manageable.

## Notes On The Model

The main regression treats each card as a binary indicator showing whether it appears in a player's deck. Win rate is the response variable.

This approach is useful for estimating average marginal effects of cards, but it has clear limits:

- It ignores interaction effects between cards.
- It does not directly account for player skill, matchup quality, or level differences beyond what is already reflected in win rate.
- It can recommend unrealistic deck combinations because statistical fit is not the same thing as deck synergy.

That limitation is part of the point of the project: the model is informative, but practical gameplay still matters.
